## Why this journey

I started learning ROS2 with a generic Korean curriculum that taught Topic / Service / Action,
URDF, TF, and so on. After two weeks I realized something that bothered me:

> "I know how nodes talk. I know how to write a Publisher. But what does each different
> robot type *actually mean* in terms of motion? Why is one called Differential and
> another Mecanum?"

The textbook just says "two wheels" or "four mecanum wheels" without making the kinematic
difference *physical*. So I decided to take a different path: instead of building one custom
robot via URDF (the standard curriculum), I would **drive five different existing robots in
simulation and feel each kinematic type with my own hands**.

This blog is the record of that journey.

---

## The plan

```
Phase 1: Differential Drive       (2 wheels, independent speed)
   ↓ add omnidirectional capability
Phase 2: Mecanum / Omnidirectional (4 wheels, diagonal rollers)
   ↓ add steering, lose omnidirectional
Phase 3: Ackermann (FWD + steer)   (2 driving wheels, both front, steered)
   ↓ add 4-wheel drive
Phase 4: Ackermann + Traction      (front steer + 4-wheel independent speed)
   ↓ add rear steering
Phase 5: Swerve Drive              (4 modules, each independently steered + driven)
```

Each phase adds **exactly one new degree of freedom** over the previous one. By the end,
the difference between a $200 differential drive robot and a $50,000 swerve research
platform becomes intuitive instead of academic.

## Why Neobotix

I needed three different kinematic models. The cheapest path was the **Neobotix neo_simulation2**
package, which has:

- **MP-400**: differential drive (matches Phase 1)
- **MPO-500**: mecanum drive (matches Phase 2)
- **MPO-700**: omni-drive modules = swerve! (matches Phase 5)

And by **artificially constraining how I use MPO-700**, I could simulate Phase 3 and Phase 4
on the same robot. That gave a clean consistency: Phase 3, 4, 5 all use the same MPO-700,
just with different control strategies.

## The 7-step learning cycle (used every phase)

```
1. Kinematics theory      Read the equations, understand the variables
2. Install simulation     Get the robot running in Gazebo + RViz
3. Observe stationary     Look at TF tree, measure wheelbase / track
4. Manual control         Send cmd_vel by hand, watch how it responds
5. Code automation        Write a Python node that drives a pattern
6. Compare to previous    What can this robot do that the previous one couldn't?
7. Notes for next phase   What's still missing? What will the next phase add?
```

This cycle gives consistency across phases. After Phase 2 I could move through the cycle
in 2-3 days; for Phase 5 it took about a week because of the math.

## What's special about this approach

- **Same cmd_vel interface, different internal behavior.** All five robots take
  `geometry_msgs/Twist` but interpret it completely differently. You can't tell from
  the topic API which kinematic type you're talking to. You have to know.

- **Constraint as learning device.** Phase 3 and Phase 4 use the same MPO-700 swerve
  robot but I voluntarily refuse to use `linear.y`. Holding back from the full
  capability is what teaches you what each constraint costs.

- **Hand-coded kinematics calculator.** For Phase 4 and 5, even though the simulator
  computes things internally, I wrote my own Python module that calculates *what each
  wheel should be doing* given a `cmd_vel`. This is the difference between using
  kinematics and knowing kinematics.
